# `htt repl`: interactive todo-mode shell

**Date:** 2026-05-16
**Status:** Proposed

## Motivation

The CLI is great for one-off commands and the TUI is great for
arrow-key task management. Neither matches the "I'm sitting down to
process my todo list for the next ten minutes" workflow, where the
user wants to:

- See the current context's tasks
- Issue a few commands in succession (add, complete, prioritise,
  switch context)
- See the effect of each command on the task list immediately
- Not retype `htt todo` on every line

A REPL fills the gap. Each cycle clears the screen, renders the
current context, and waits for one command. The command runs, its
effect is reflected on the next render. The user types in
todo-mode (`add foo`, `delete 0`, `+ 1`); a `/` prefix escapes to
the full command path (`/log start review`, `/report --since 7d`).

This is small enough to ship as a single new file with no impact
on the rest of the codebase.

## Decisions captured

From the planning dialogue (2026-05-16):

1. **Todo-mode by default, `/` to escape.** Inputs without a leading
   `/` are dispatched as `todo <input>`. Inputs starting with `/`
   have the slash stripped and run as full CLI commands. This makes
   the common case terse without losing access to the rest of the
   surface.

2. **Clear + redraw each cycle.** Each iteration clears the
   terminal, renders the current context (same shape as `htt todo
   show`), then shows the prompt. Command output prints briefly
   after Enter, visible until the user types the next command —
   that's the only persistence window for diagnostics. The view
   itself reflects the effect of the previous command, so most
   diagnostics are redundant.

3. **Todo subcommands only as the default surface.** Without a `/`
   prefix, only the `todo` subcommand tree is reachable
   (`add`, `delete`, `do`, `priority`, `+`, `-`, `move`, `replace`,
   `context`, `show`, `random`, `search`, `edit`, `edit-done`,
   `edit-archive`). Everything else needs `/`.

4. **Exit on Ctrl-D, Ctrl-C, `quit`, `exit`, or `q`.** All five
   exit cleanly. Ctrl-C terminating the program (rather than
   cancelling the current line) is the readline default and we
   keep it — easier than fighting the lib.

5. **Prompt format `htt(<context>)> `.** The `htt` prefix makes
   the mode explicit; the parenthesised context name updates after
   `context <name>` switches.

6. **Library: `chzyer/readline`.** Gives command history (up/down)
   and basic line editing for one extra dep. Doesn't natively
   support fixed-position prompts — accepted tradeoff. If the
   ergonomics push back hard on this, a future revision could
   rebuild the REPL as a Bubble Tea program (sibling to the
   existing TUI). Not now.

7. **Empty Enter refreshes the view.** Most useful after an
   external edit (file change in another terminal, sync pull). The
   alternative — bare re-prompt — is the typical shell behaviour
   but less useful here.

8. **Non-TTY (piped) input is not a goal.** Readline degrades when
   stdin isn't a TTY; we don't test that path. Scripting goes
   through the CLI directly.

## Shape

A single new file: `internal/cli/repl.go`. New Cobra command
registered on `RootCmd`. No changes to existing files beyond the
init-block registration.

```go
// internal/cli/repl.go

var Repl = &cobra.Command{
    Use:   "repl",
    Short: "Interactive todo-mode shell.",
    Args:  cobra.NoArgs,
    RunE: func(cmd *cobra.Command, args []string) error {
        return runRepl()
    },
}

func init() {
    RootCmd.AddCommand(Repl)
}

func runRepl() error {
    // 1. Set up readline with history file under config dir.
    // 2. Loop:
    //    a. Clear screen.
    //    b. Render current context (call ctx.Show()).
    //    c. Read line via readline.
    //    d. Handle exit words / empty / "/"-prefix / default.
    //    e. Dispatch via RootCmd.SetArgs + Execute.
    //    f. Errors from Execute print but don't exit the REPL.
}
```

### Dispatch detail

For an input line `s` with leading/trailing whitespace stripped:

| Form              | Dispatch                                                    |
| ----------------- | ----------------------------------------------------------- |
| `""`              | No-op — loop iterates, view refreshes                       |
| `quit`/`exit`/`q` | `return nil`                                                |
| `/<rest>`         | Split `<rest>` on whitespace, `RootCmd.SetArgs(...)`        |
| `<anything>`      | Split, prepend `["todo"]`, `RootCmd.SetArgs(...)`           |

Splitting uses simple whitespace tokenization (Go's
`strings.Fields` — entries with literal spaces inside go through
the natural `add ship the auth refactor` join, same as CLI today).

### Prompt format

```
htt(work)>
```

The context name comes from `repo().CurrentContextName()` and
refreshes after every command (since `context <name>` may have
switched it).

### History

`readline.Config{ HistoryFile: ... }` points at a file under the
config dir (`<configDir>/repl-history`). Persisted across
sessions; up/down arrow walks it. No history-search shortcuts
needed for v1.

### Output behaviour

Cobra commands print to stdout/stderr as usual. We don't capture
or redirect — output flows to the terminal. The next iteration's
clear wipes it. The view re-render reflects the effect.

Errors returned from `RootCmd.Execute()` print to stderr (the
prevailing pattern for Cobra) but the REPL loop continues. Only
EOF / signals / explicit exit words break the loop.

## Steps

A single commit. Tests where they add real coverage; no harness
gymnastics for a thin shell.

### Step 1 — `internal/cli/repl.go`

1. New file with `Repl` Cobra command + `runRepl` loop as
   sketched above.
2. Use `chzyer/readline` for line input. Configure history file
   path (under `viper.ConfigFileUsed()` directory or
   `os.UserConfigDir()`).
3. Helper to render the view: `repo().CurrentContext()` →
   `ctx.Show()`.
4. Dispatch helper: parse the input form (`""` / `quit` /
   `/<rest>` / default) and call `RootCmd.SetArgs` +
   `RootCmd.Execute`.
5. Register `Repl` on `RootCmd` in the `init` block.

### Step 2 — Tests

Two unit tests in `internal/cli/repl_test.go`:

- `TestRepl_DispatchClassification`: a pure-function helper that
  classifies an input string into `{exit, none, args []string}`,
  table-tested:
  - `""` → exit=false, none=true
  - `"quit"` / `"exit"` / `"q"` → exit=true
  - `"/log start review"` → args=`["log", "start", "review"]`
  - `"add buy milk"` → args=`["todo", "add", "buy", "milk"]`
  - `"  add buy milk  "` (whitespace) → same as above
- `TestRepl_DispatchExecutesCommand`: assemble a memory repo,
  call the dispatch function with `"add buy milk"`, verify the
  task lands in the repo.

The clear-screen / readline-loop logic stays uncovered by tests —
it requires a TTY and the value-per-test-cost is poor. Acceptable
gap for a thin shell.

### Step 3 — Docs

- Add a short "Interactive mode (REPL)" section to
  `GETTING-STARTED.md` mentioning `htt repl`, the `/` escape, and
  the exit words.
- Drop nothing from `TODO.md` — REPL wasn't on the wishlist
  explicitly, this is net-new.

## Out of scope

- **Fixed-position prompt (anchored top/bottom of screen).**
  Requires owning the screen; would force the REPL into Bubble
  Tea territory. Revisit only if the line-stream REPL ergonomics
  are bad in daily use.
- **Tab completion / autocomplete.** Readline doesn't ship with
  Cobra-aware completion; wiring it would add a fair bit of
  plumbing for a personal-use tool. Defer.
- **Multi-line input** (e.g., heredocs for long task entries).
  Tasks are one-liners by todo.txt convention; if a user wants
  a paragraph, `htt todo edit <id>` shells to `$EDITOR`.
- **Scripting via piped stdin.** Non-TTY is degraded by readline
  and not a target. The CLI is the scripting interface.
- **REPL launches the TUI / nested REPL.** `/interactive` would
  technically work (Cobra dispatches it), but the result is two
  programs fighting for the terminal. Disable or pass-through-
  with-warning; pin behaviour during implementation.
